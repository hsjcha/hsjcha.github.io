写在前面

这是我正式写下来的第一篇博客。

一直以来，我都在零零散散地学习一些和安全相关的内容，比如 Web 漏洞、抓包分析、CTF 题目和脚本编写，但很多时候只是“学过了”，却没有真正留下记录。时间一久，曾经踩过的坑、看过的知识点、做题时的思路，都会慢慢变得模糊。

所以我想从这篇文章开始，把学习过程认真地记录下来。一方面是为了帮助自己整理思路，另一方面也是希望未来回头看的时候，能清楚地看到自己是怎么一步一步走过来的。

为什么开始学安全

最开始对安全感兴趣，其实是因为我觉得它很“真实”。

和单纯地学习语法、框架不同，安全更像是在理解一个系统是怎么被搭起来的，又可能会在哪里出问题。你不仅要知道功能是怎么实现的，还要去思考：

  • 用户输入会不会被错误地信任
  • 参数传递的边界在哪里
  • 一段看起来正常的逻辑，是否可能被恶意利用

这种从“开发视角”切换到“攻击与防御视角”的过程，对我来说很有吸引力。它会逼着我去更深入地理解程序本身，而不是只停留在“会用”。

我目前在学什么

现阶段我主要把精力放在两个方向：

1. Web 安全基础

这是我最先接触、也最容易建立反馈的方向。比如：

  • SQL 注入
  • XSS
  • 文件上传
  • 命令执行
  • 身份认证与权限控制问题

这些内容看起来是“漏洞类型”，但本质上是在训练一种思维方式：当一个功能被设计出来后，系统默认相信了什么，又遗漏了什么。

2. CTF 题目练习

CTF 对我来说很像一种高强度的实战训练。它不会给你完整的背景，也不会一步一步告诉你怎么做，而是需要你自己从题目给出的信息中找线索、做判断、写脚本、验证结果。

这个过程虽然经常会卡住，但也正是最能锻炼问题分析能力的部分。

我希望这类博客记录什么

从今天开始，这个博客会更偏向“学习笔记 + 过程复盘”，我会尽量记录下面这些内容：

  • 某个漏洞原理的理解
  • 做一道题时的思考过程
  • 工具使用中的细节和踩坑
  • 自己写的小脚本与自动化思路
  • 看文章、看源码后的关键收获

我不希望它只是把结论抄一遍，而是尽量把“我是怎么想到的、为什么会卡住、最后怎么解决的”写出来。因为很多时候，真正有价值的不是答案本身，而是推到答案的路径。

给现在的自己留一句话

安全学习很容易让人产生挫败感。

有时候一个环境搭半天跑不起来，有时候一道题看了很久都没有思路，有时候明明知道漏洞名字，却不知道真正落到代码里应该怎么看。这个过程并不轻松，但我也慢慢意识到，很多成长恰恰来自这些“不顺利”的时刻。

所以这篇文章也算是一个小小的起点。先不要求自己一下子写得多专业，也不要求每篇都很完整，先把记录这件事坚持下来。

结语

这篇博客不算一篇“技术很深”的文章，但它对我来说很重要。

它意味着我不再只是零散地学习，而是开始有意识地整理、复盘和输出。希望未来的自己看到这篇文章时，会感谢现在这个愿意认真开始的自己。

如果你也刚好在学习安全，那就一起慢慢进步。
