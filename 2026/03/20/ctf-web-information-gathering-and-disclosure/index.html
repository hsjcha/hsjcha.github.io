很多新手做 CTF Web 时，最容易卡住的点不是“不会漏洞”，而是不知道该从哪里看、先看什么、看到一个线索后又该往哪条路继续走。

这篇文章就只做一件事：把 信息收集 和 信息泄露 这件事讲透，而且尽量只讲对做题真正有用的部分。你可以把它理解成一条从“看到目标”到“找到入口”的路线图。

先建立一个做题脑图

对 CTF Web 来说，信息收集的目标不是“知道得越多越好”，而是尽快回答下面几个问题：

  • 这个站点还有没有别的入口
  • 这个入口背后跑的是什么技术栈
  • 有没有隐藏页面、隐藏参数、隐藏接口
  • 有没有已经泄露出来的源码、配置、账号、路径、报错信息
  • 这些线索最终能不能转成漏洞利用点

如果你一上来就只会盲扫目录、盲打 payload，很容易陷入“看起来很努力，但实际上没有方向”的状态。更高效的方式是按顺序走：

  1. 先做被动信息收集，尽量不与目标产生太多交互。
  2. 再做主动探测，确认入口、端口、目录、指纹和参数。
  3. 最后把拿到的线索映射成具体漏洞方向。

你可以先记住一句话：

信息收集的本质，是把目标的攻击面从“一个网址”还原成“一整套可分析的系统”。

一条适合新手的 CTF Web 工作流

如果题目只给了一个站点或一个域名，我建议按下面这个顺序推进：

  1. 浏览器访问首页，观察标题、报错、跳转、注释、脚本、请求包。
  2. 看响应头、Cookie、Server、X-Powered-By、状态码、重定向位置。
  3. 查常见泄露点，如 robots.txt、sitemap.xml、phpinfo.php、.git/HEAD、备份文件。
  4. 做目录和文件枚举，找后台、上传点、调试页、接口文档、备份包。
  5. 做指纹识别，确认语言、框架、CMS、中间件、WAF 特征。
  6. 如果题目是域名而不是单站，再做子域名、GitHub、证书透明度、网络空间测绘。
  7. 如果题目给的是 IP、网段或者明显是多主机场景，再考虑端口扫描和 C 段收集。

这个顺序的核心原因很简单：CTF Web 题往往不是让你“扫得最全”，而是让你尽快定位“最值得打的点”。

被动信息收集：先把公开线索拿干净

被动信息收集的好处是快、隐蔽、成本低。在真实攻防里它能减少暴露，在 CTF 里它能帮你少走很多弯路。

1. 搜索引擎不是用来搜文章的，是用来搜入口的

搜索引擎最重要的价值，不是“搜到官网”，而是搜到那些开发者没打算给普通用户看的页面。

最常用的几个语法：

这些语法在做题里最常拿来找下面几类东西：

  • 后台登录页
  • 文件索引目录
  • 文档、备份、配置文件
  • 调试页面和测试页面
  • 敏感关键词关联页面

做题时要特别留意这几种搜索结果：

  • 路径名很像后台，如 /admin、/manage、/console、/login
  • 页面标题暴露系统名、版本号、组件名
  • 搜到 Index of，说明目录浏览可能开着
  • 搜到 .git、.svn、.bak、.zip、.sql，优先级很高
  • 搜到 PDF、DOCX、XLSX，可能带路径、邮箱、子域名、用户名

很多人学搜索语法时容易犯一个误区：只记语法，不知道为什么搜。真正有用的思路是反过来想：

  • 我想找后台，那就搜 inurl:admin
  • 我想找泄露文件，那就搜 filetype: 和备份后缀
  • 我想找目录遍历，那就搜 "Index of"
  • 我想找开发痕迹，那就搜系统名、项目名、框架名

2. GitHub 往往泄露的不是代码本身，而是上下文

很多 CTF Web 题不会直接把源码给你，但会通过各种方式让你接近源码。GitHub 搜索就是常见入口之一。

常见搜索思路：

在 GitHub 里你最想看到的东西不是“炫技代码”，而是下面这些：

  • 配置文件
  • 数据库连接串
  • API Key、Token、Secret
  • 上传路径、静态目录、调试开关
  • 提交历史里的旧密码、旧接口、旧路径
  • 代码注释里的接口说明和业务逻辑

对 CTF Web 来说，GitHub 线索最常带来三类收益：

  • 确认技术栈，比如 Flask、Laravel、ThinkPHP、Spring Boot
  • 拿到敏感配置，比如数据库账号、JWT 密钥、Redis 地址
  • 找到隐藏接口和参数名，比如 /api/debug、/upload、role、is_admin

如果你已经拿到了站点名、页面标题、JS 文件名、报错堆栈中的目录名，就更应该去 GitHub 搜这些关键词。很多题目的“线索链”就是这样拼起来的。

3. 网络空间测绘平台适合做“资产扩展”

像 FOFA、Shodan、ZoomEye 这类平台，本质上是在帮你回答一个问题：

这个目标在互联网里，还暴露了哪些同类资产？

常见检索维度包括：

  • domain
  • ip
  • port
  • title
  • header
  • body
  • banner
  • server

在 CTF Web 中，这类平台最有价值的场景有两个：

  • 题目给的是域名，你怀疑还有别的子站、别的端口、别的服务
  • 题目给的是一个登录页或组件页，你想反推出系统版本和部署特征

4. 子域名收集的意义，是把“一个站”扩展成“一片站”

很多新手看到一个目标域名，就默认“所有内容都在这个域名的首页里”。这通常是错的。

子域名常常意味着：

  • 不同业务系统
  • 不同开发团队
  • 不同权限体系
  • 不同技术栈
  • 不同安全成熟度

举个非常常见的情况：

  • www.example.com 是正式站
  • admin.example.com 是后台
  • api.example.com 是接口
  • dev.example.com 是测试环境
  • static.example.com 放前端资源

对于 CTF Web，子域名里最值得关注的往往不是“数量多”，而是下面这些名字：

  • admin
  • api
  • dev
  • test
  • uat
  • staging
  • oa
  • git
  • upload

工具上可以知道有 OneForAll、Subfinder、Amass 这一类，但你要明白重点不在于“我会不会某个工具”，而在于：

  • 发现子域名后要立刻访问
  • 看是否共用 Cookie 或单点登录
  • 看是否存在开发环境、未授权接口、调试页面
  • 看是否能从一个低权限子域影响主站

如果是纯单站题，这一步的重要性不一定高；如果是多域名、多主机题，这一步往往直接决定你能不能进下一层。

主动信息收集：开始和目标交互

被动信息收集只能帮你拿到公开线索，真正做题时，很多关键信息还是得靠你自己发请求、看响应、做枚举。

1. 浏览器和 Burp 是最先该打开的工具

别一上来先跑扫描器。先手工看页面，通常更快。

你至少要看这几样：

  • 页面源码里有没有注释、隐藏字段、被注释掉的接口
  • JS 里有没有接口地址、参数名、鉴权逻辑、路由路径
  • 网络请求里有没有额外接口，如 /api/user、/graphql、/debug
  • Cookie 里有没有明显的身份信息，如 role=user
  • 响应头里有没有 Server、X-Powered-By、Set-Cookie
  • 出错时有没有文件绝对路径、版本号、框架报错页

很多题目不是靠“扫”出来的，而是靠下面这种细节看出来的：

  • 前端只校验长度，后端没校验
  • 页面没入口，但 JS 里写了接口
  • 隐藏参数 debug=1
  • 管理员页面只是前端按钮被隐藏
  • 同一个接口对不同请求方法响应不同

如果你能把 Burp Suite 和浏览器开发者工具用熟，很多 CTF Web 题的前期效率会直接翻倍。

2. 目录与敏感文件枚举，是最直接的“找入口”方式

这一部分几乎是所有 Web 题都会碰到的。

优先检查这些路径：

  • /robots.txt
  • /sitemap.xml
  • /favicon.ico
  • /admin
  • /login
  • /upload
  • /backup
  • /test
  • /phpinfo.php
  • /.git/HEAD
  • /.svn/entries

再配合目录扫描工具去扩展，比如 dirsearch、gobuster、ffuf。

做目录扫描时最重要的不是“扫得快”，而是“会判断结果值不值得继续”。

你要重点盯这几类返回：

  • 200：直接存在
  • 301/302：可能跳到真实路径
  • 401：有鉴权，说明后台可能真实存在
  • 403：被禁止访问，但不代表不存在

尤其是 403，新手很容易忽略。实际上，403 往往意味着：

  • 路径是真的
  • 只是当前请求方式不对
  • 或者来源、认证、IP、Header 不满足要求

3. 端口扫描解决的是“站点背后还有什么”

如果题目给的是 IP，或者你怀疑目标不是单纯的 80/443 网站，就该看端口。

端口扫描的目的不是为了记住一堆默认端口，而是为了推断：

  • 还有没有别的 Web 服务
  • 有没有后台中间件
  • 有没有数据库、缓存、消息队列误暴露
  • 有没有能配合 Web 利用的服务

最常见的例子：

  • 80/443：HTTP/HTTPS
  • 8080：常见备用 Web 端口
  • 7001/7002：WebLogic
  • 8000/8081/8888：测试站、管理台、开发服务常见
  • 3306：MySQL
  • 6379：Redis
  • 9200：Elasticsearch

在 CTF Web 里，看到额外端口时要立刻想三个问题：

  1. 这是不是另一个入口。
  2. 这是不是能提供版本和组件信息。
  3. 这是不是能反过来帮助我利用主站。

4. 指纹识别是在猜“这个站会栽在哪类坑里”

指纹识别的价值不只是知道“它是 WordPress”或者“它跑了 PHP”，而是把漏洞方向缩小。

你想识别的东西包括：

  • Web 服务器，如 Nginx、Apache
  • 后端语言，如 PHP、Python、Java
  • 框架，如 Flask、Django、Spring Boot
  • CMS，如 WordPress、Drupal
  • 前端框架，如 Vue、React
  • WAF 或 CDN 特征

常见工具：

  • WhatWeb
  • Wappalyzer
  • Nmap -sV

  • PHP + Apache，就多想文件包含、源码泄露、弱类型、上传点
  • Python Flask，就多想模板注入、调试模式、签名伪造
  • Java Spring，就多想接口暴露、Actuator、反序列化、SpEL
  • WordPress，就多看插件、主题、后台路径、备份包

指纹 不直接等于 漏洞，但它能帮你少走很多错误方向。

5. C 段收集什么时候才重要

原始材料里有 C 段收集，这个知识点本身没问题，但对多数单站 CTF Web 题来说不是最高优先级。

它更适合下面这些场景：

  • 题目给的是一个网段
  • 靶场是内网渗透联动场景
  • 你需要从一台入口机继续发现同网段资产
  • 题目明显是“多主机、多服务”设计

如果只是一个普通网页题，先去找参数、接口、目录、泄露，比直接扫整个网段更有效。

信息泄露：很多题目的真正入口

对新手来说，“信息泄露”最容易被低估。很多人以为它只是“顺手看一眼”，但实际上不少题目的突破口就是泄露本身。

1. 测试页面和调试页面

优先检查：

  • /test.php
  • /info.php
  • /phpinfo.php
  • /debug
  • /dev
  • /install
  • /setup

其中 phpinfo() 页面特别危险，因为它可能直接告诉你：

  • PHP 版本
  • 网站根目录
  • Session 保存路径
  • 已加载模块
  • 服务器环境变量
  • 临时目录和上传目录

这些信息在题目里经常可以继续转化成：

  • 文件包含利用
  • 上传绕过
  • Session 相关利用
  • 已知组件漏洞定位

2. 源码泄露和版本库泄露

这部分优先级非常高。

你要检查的对象包括：

  • /.git/HEAD
  • /.git/config
  • /.svn/entries
  • 网站备份压缩包
  • 历史源码包
  • 误传的仓库目录

常见后缀：

  • .zip
  • .rar
  • .7z
  • .tar.gz
  • .bak
  • .old
  • .swp
  • .tmp

为什么源码泄露这么重要？因为它能直接回答很多做题里最难的问题：

  • 参数到底怎么校验
  • 后端到底查了什么数据库
  • 登录逻辑到底怎么判断权限
  • 文件上传白名单和黑名单是什么
  • 模板渲染点在哪里
  • 有哪些你在前端没看到的隐藏路由

一句话总结：

拿到源码后，很多 Web 题就从“猜”变成了“读”。

3. 配置文件泄露

配置文件里常见的高价值信息包括：

  • 数据库账号密码
  • Redis、MongoDB、Elasticsearch 地址
  • JWT 密钥
  • Session 配置
  • 第三方接口 Token
  • Debug 开关

常见目标文件：

  • .env
  • config.php
  • application.properties
  • application.yml
  • web.config
  • nginx.conf
  • .htaccess
  • settings.py
  • config.yaml

在 CTF 里，这类文件的价值经常不止是“拿凭证”，还可能让你直接发现：

  • 文件上传目录
  • 模板目录
  • 数据库表名
  • 管理员用户名
  • 内网服务地址

4. 日志、临时文件、编辑器残留

这类东西经常被忽略，但命中率不低。

例如：

  • 日志文件可能有报错堆栈和 SQL 语句
  • .swp 可能有编辑中的源码片段
  • .DS_Store 可能暴露目录结构
  • *.log 可能暴露后台接口或用户行为
  • 备份的旧文件可能没有新版本的权限校验

如果一个题看起来“主逻辑防得挺严”，那旧文件、残留文件、临时文件往往就是更容易下手的地方。

5. 前端泄露其实非常常见

前端本身也是信息泄露的重要来源。

重点看：

  • HTML 注释
  • 隐藏字段
  • 被注释掉的按钮或路由
  • JS 中硬编码的接口地址
  • source map 文件，如 app.js.map
  • 前端写死的用户名、路径、Token、调试开关

很多题目会把线索藏在下面这种位置：

  • fetch("/api/admin/list")
  • isAdmin === true
  • role: "guest"
  • debug: false
  • /static/backup.zip

对新手来说，一个很重要的转变是：

不要把前端当成“样式层”，要把前端当成“情报层”。

怎么把线索转成漏洞方向

真正决定你能不能做出来的，不是看到了多少信息，而是你能不能把信息和漏洞类型对应起来。

下面这张映射关系很值得背熟：

  • 看到后台路径、管理接口、权限字段。
    想到未授权访问、越权、弱口令、参数篡改。

  • 看到上传目录、文件后缀校验、图片处理逻辑。
    想到文件上传、解析漏洞、后缀绕过、路径穿越。

  • 看到报错堆栈、SQL 报错、模板报错。
    想到 SQL 注入、模板注入、路径泄露、源码路径定位。

  • 看到 .git、备份包、源码片段。
    想到读代码找鉴权缺陷、隐藏接口、硬编码密钥。

  • 看到框架和版本。
    想到已知漏洞、默认路径、默认口令、组件特征利用。

  • 看到 API 文档、swagger、graphql、调试接口。
    想到未授权接口、批量枚举、参数污染、业务逻辑缺陷。

  • 看到 phpinfo()、环境变量、绝对路径。
    想到文件包含、上传临时文件、Session 文件、日志文件利用。

换句话说，信息收集不是漏洞利用之前“可有可无的准备动作”，它本身就在不断缩小漏洞范围。

一套对新手很有用的检查清单

如果你做一题时总不知道下一步该干什么，可以按这个清单过一遍：

第一步：先看页面本身

  • 看标题、注释、源码、隐藏字段
  • 抓包看请求头、响应头、Cookie、状态码
  • 看是否有跳转、报错、版本号、路径泄露

第二步：找入口

  • 访问 robots.txt
  • 访问 sitemap.xml
  • 访问常见后台和调试路径
  • 扫描目录、文件、备份后缀

第三步：找技术栈

  • 看 Server 和 X-Powered-By
  • 跑 WhatWeb 或浏览器插件
  • 从页面特征、文件扩展名、报错页判断框架

第四步：扩展攻击面

  • 如果是域名，查子域名
  • 如果是 IP，扫端口
  • 如果是多主机题，考虑 C 段和旁站
  • 如果有项目名，去查 GitHub

第五步：盯泄露

  • 看 .git、.svn、备份压缩包
  • 看测试页、phpinfo.php、调试页
  • 看配置文件、日志文件、临时文件
  • 看 JS、source map、接口文档

第六步：把线索转利用

  • 有路径就想目录穿越、文件包含、未授权
  • 有账号密码就想登录、复用、越权
  • 有源码就读鉴权、读过滤、读上传、读模板
  • 有版本就查默认特征和已知漏洞方向

新手最常见的几个误区

  • 只扫目录，不看返回内容。很多真正的线索在响应体里，不在状态码里。
  • 看到 403 就放弃。403 常常意味着路径真实存在。
  • 只盯首页，不看 JS 和接口请求。很多题目关键入口在前端脚本里。
  • 只想漏洞名，不想业务链。很多题不是“某个高危漏洞”，而是“一个小泄露加一个小逻辑缺陷”。
  • 不会读源码。拿到源码后如果还只会继续扫，那就浪费了最有价值的线索。

最后把这件事想简单一点

对 CTF Web 来说，信息收集并不神秘，它只是一个不断回答问题的过程：

  • 目标还有什么我没看到
  • 这些东西藏在哪里
  • 它们说明后端在干什么
  • 哪些信息已经不小心泄露出来了
  • 这些泄露能不能帮我打到真正的漏洞点

当你开始习惯用这条思路做题后，你会发现很多原来看起来“无从下手”的题，其实并不是没漏洞，而是你还没有把攻击面展开。

如果你是刚入门，建议你先把这篇文章里的流程反复练熟。因为对绝大多数 CTF Web 题来说，先把信息收集做对，后面的利用往往就会顺很多。
