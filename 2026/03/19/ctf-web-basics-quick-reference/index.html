这篇文章的目标不是“讲全”，而是先把做 CTF Web 时最常用、最容易卡住的基础点收成一篇能快速翻的速查笔记。

先记住：Web 题本质上在看什么

大多数 CTF Web 题，绕不开下面几件事：

  • 请求是怎么发出去的
  • 参数是怎么传到后端的
  • 数据在中间经历了什么编码和解码
  • 前端有没有泄露线索
  • 后端语言怎么接收和处理输入
  • 登录态、身份信息、权限信息存在哪里
  • 靶机环境到底是主机直跑，还是跑在容器里

如果这几件事的链路能看清，很多题就不会只剩“盲试 payload”。

HTTP、URL 与请求响应

Web 题先别急着打 payload，先把请求结构看明白。

做题时最该注意的是：

  • scheme：http 和 https，决定是否走 TLS。
  • host：域名或 IP，有时题目会藏子域名、虚拟主机。
  • port：不写时 HTTP 默认 80，HTTPS 默认 443。
  • path：资源路径，经常决定路由、文件位置、接口入口。
  • query：问号后面的参数，很多题的输入点就在这里。
  • fragment：# 后的内容不会发给服务器，只在浏览器端使用。

最基础的请求/响应模型可以理解成：

  1. 浏览器解析 URL。
  2. 建立连接并发请求。
  3. 服务端处理后返回响应。
  4. 浏览器继续渲染 HTML、加载 CSS、JS、图片等资源。

做题时要重点看这几个位置：

  • 请求方法：GET、POST、PUT、DELETE
  • 请求头：Host、Cookie、Content-Type、User-Agent、Referer
  • 请求体：表单、JSON、上传文件
  • 响应头：Set-Cookie、Location、Content-Type
  • 状态码：200、302、403、404、500

在 CTF Web 里，很多信息都藏在“请求哪里不一样”：

  • 参数名是不是可控
  • 参数值有没有被过滤
  • 响应长度有没有变化
  • 状态码是不是跳转了
  • 有没有多出来的 Set-Cookie

编码与进制：很多题不是漏洞难，是你没先解码

看到奇怪字符串，先别慌，优先怀疑编码。

最常见的几种：

  • URL 编码：< 变成 %3C，空格常见为 %20
  • HTML 实体：< 变成 &lt;
  • Base64：常见由大小写字母、数字、+、/、= 组成
  • Unicode：如 \\u0061 表示 a
  • Hex：十六进制，如 61 62 63
  • ASCII / UTF-8：字符和字节之间的映射

做题里经常要会这几个动作：

  • 识别“这是原始数据还是编码后的数据”
  • 判断“被编码了几次”
  • 知道“服务端在哪一层会解码”

它们分别可能对应：

  • URL 编码后的 XSS 负载
  • Base64 编码后的 admin:123456
  • Unicode 转义后的 PHP 代码片段
  • 十六进制表示的字符串

进制本身不复杂，但 CTF Web 很常见：

  • 十六进制看字符串：0x616263 就是 abc
  • ASCII 对照回显：十进制、十六进制、字符互转
  • 二进制和位运算偶尔会出现在验证码、加密、校验逻辑里

如果页面、接口、参数看起来“像乱码”，优先做下面的检查：

  • 有没有 %xx
  • 有没有 \\u
  • 有没有一长串只由 Base64 字符集组成的内容
  • 有没有十六进制前缀 0x
  • 解码一次后是不是还能继续解

HTML、CSS、JavaScript：前端不只是样式，还是线索区

很多新手看到前端代码就跳过，但 Web 题里，前端经常直接送分。

看页面源码时，建议重点看：

  • 表单提交到哪里：action
  • 提交方式：method
  • 隐藏字段：input type="hidden"
  • 注释里的提示
  • 外链 JS 里的接口路径
  • 前端写死的账号、路径、参数名
  • fetch、ajax、axios 请求逻辑
  • 是否有前端校验但后端未必校验

  • /api/debug
  • /admin
  • /flag
  • 被注释掉的调试代码
  • 前端校验逻辑
  • Base64、MD5、AES 之类的前端处理过程

一句话概括：前端代码不一定决定漏洞，但它经常决定你该从哪里下手。

PHP：先会看输入从哪进来，再看它怎么被处理

PHP 是 CTF Web 里非常高频的后端语言。基础语法不用背太多，先抓最关键的入口。

最常见的超全局变量：

  • $_GET：取 URL 参数
  • $_POST：取请求体参数
  • $_REQUEST：可能同时包含 GET、POST、COOKIE
  • $_COOKIE：取 Cookie
  • $_SESSION：取会话数据
  • $_FILES：处理文件上传

除了变量入口，PHP Web 题还经常围绕这些点展开：

  • 字符串拼接
  • 文件包含
  • 文件读取
  • 命令执行
  • 反序列化
  • 上传与解析
  • 弱比较

常见危险函数和敏感点，建议先认识名字：

  • eval
  • system
  • exec
  • passthru
  • include
  • require
  • file_get_contents
  • unserialize
  • md5
  • strcmp

这里不展开每个漏洞，只强调一个做题思路：

  • 先找输入点
  • 再找过滤点
  • 再找危险操作点
  • 最后看数据能不能从输入流到危险点

也就是典型的“source -> sanitize -> sink”。

Cookie 与 Session：很多登录题，本质是状态题

HTTP 本身无状态，所以网站需要额外机制记住“你是谁”。

最常见的就是：

  • Cookie：状态标识存浏览器
  • Session：状态数据存服务端，浏览器通常只存一个 Session ID

这里最重要的不是死记属性，而是知道它们在题里意味着什么：

  • PHPSESSID=abc123：这是身份标识，不一定是身份本身
  • HttpOnly：前端 JS 不容易直接读到
  • Secure：通常只在 HTTPS 里发
  • Path、Domain：决定 Cookie 作用范围

做 Session 相关题时，经常要问自己：

  • 登录成功后，多了哪个 Cookie
  • 修改 Cookie 后，权限有没有变化
  • 是不是把身份信息直接放在 Cookie 里了
  • 后端是不是只校验了“有无 Session”，没校验“Session 对应谁”
  • URL 参数里有没有直接带会话标识

如果题目涉及登录、管理员、购物车、验证码、个人中心，基本都值得先抓包看看：

  • 有没有 Set-Cookie
  • 有没有 remember me
  • 有没有 JWT
  • 有没有明显的用户编号、角色编号

Docker：很多 Web 靶机其实跑在容器里

现在很多题目环境不是直接装在系统上，而是跑在 Docker 容器里。对做题的人来说，不需要一开始就把 Docker 学得很深，但至少要知道它会影响你怎么看端口、源码、日志和服务关系。

先记住 3 个概念：

  • image：镜像，可以理解成“运行模板”
  • container：容器，是镜像跑起来后的实例
  • volume / bind mount：挂载，把宿主机目录映射进容器

如果一道题给了源码、Dockerfile、docker-compose.yml 或者启动脚本，那它通常在告诉你两件事：

  • 服务是怎么跑起来的
  • 关键文件和端口在哪里

做 CTF Web 时最常见的 Docker 线索有：

  • ports
  • volumes
  • environment
  • Dockerfile
  • docker-compose.yml
  • 多个服务名，比如 web、php、mysql、redis

它们分别常用来做什么：

  • docker ps：看当前跑了哪些容器、映射了哪些端口
  • docker logs：看报错、启动输出、调试信息
  • docker exec -it：进容器里看文件、跑命令、摸目录
  • docker inspect：看挂载、网络、环境变量、启动参数
  • docker cp：把容器里的源码、日志、配置复制出来

如果能进入容器，通常优先看这些位置：

  • Web 根目录，如 /var/www/html
  • 配置文件，如 nginx.conf、apache2.conf
  • PHP 配置和站点配置
  • 日志目录
  • /app、/srv、/usr/src/app 这种项目目录

做题时，Docker 最容易帮你解决的几个问题是：

  • “服务到底监听在哪个端口上”
  • “源码到底在容器里的哪个目录”
  • “这个文件是宿主机原生文件，还是挂载进去的”
  • “多服务之间怎么通信”
  • “环境变量里有没有账号、密码、flag 或密钥”

如果题目给了 Dockerfile，建议重点扫这几类指令：

  • FROM：基础镜像，能看出技术栈
  • COPY / ADD：源码和配置被拷到哪
  • RUN：安装了什么依赖，做了什么初始化
  • EXPOSE：容器内声明了哪个端口
  • CMD / ENTRYPOINT：容器启动时真正执行什么

如果题目给了 docker-compose.yml，建议重点看：

  • 哪些服务会一起启动
  • 哪个服务对外开放端口
  • 数据库用户名、密码、库名
  • 挂载目录对应到哪里
  • 服务之间是不是通过服务名互联

一句话概括：Docker 在 CTF Web 里不只是“部署工具”，它本身就是环境信息说明书。

一套适合新手的 Web 题排查顺序

如果你拿到一道题暂时没思路，可以先按这个顺序走：

  1. 先看页面功能，猜题目可能考什么。
  2. 打开开发者工具，看源码、JS、隐藏字段、接口路径。
  3. 抓一遍正常请求，记住参数名、请求方法、Cookie、响应头。
  4. 对输入点做最小测试，看回显、报错、状态码、响应长度是否变化。
  5. 怀疑编码时，先解码再继续看，不要直接乱打 payload。
  6. 如果后端像 PHP，就重点关注 $_GET、$_POST、$_COOKIE、$_SESSION。
  7. 如果涉及登录态，就优先研究 Cookie、Session、权限标识。
  8. 如果题目带源码或部署文件，就顺手检查 Dockerfile、docker-compose.yml 和端口映射。

很多题卡住，不是因为完全不会，而是把顺序走反了。

这篇速查最该带走的几点

  • fragment 不会发给服务器，query 会。
  • Set-Cookie 和 Cookie 是两件事，一个是服务端下发，一个是客户端回传。
  • 前端隐藏参数不等于安全参数。
  • 看不懂字符串时，优先怀疑编码，而不是优先怀疑加密。
  • 做 PHP 题，先找输入点，再找危险点。
  • 很多“权限题”本质上是“状态题”。
  • 很多靶机线索不在页面里，而在 Dockerfile、挂载和环境变量里。
